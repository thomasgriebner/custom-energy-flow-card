**Status:** v1 (proposed, ready for review)
**Datum:** 2026-05-15
**Autor:** Brainstorming-Session mit @griebner
**Verlinkte Hauptspec:** [`2026-05-10-custom-energy-flow-card-design.md`](./2026-05-10-custom-energy-flow-card-design.md)

## 0. Zusammenfassung

Die Card unterstützt aktuell ausschließlich deutsche User-facing-Strings (`src/i18n/de.ts`). Spec §1.4 listet Internationalisierung als bewussten v1.x-Kandidaten mit dem Hinweis „trivial weil zentralisiert". Diese Subspec löst genau diesen Punkt auf: eine zweite Sprache (Englisch) wird via Home-Assistant-Locale (`hass.locale.language`) automatisch aktiviert. Kein neues Config-Feld, kein User-Eingriff nötig.

Die Architektur ist eine `resolveT(lang)`-Factory in `src/i18n/index.ts`: Caller (Card, Editor, Renderer-Module) holen sich pro Render-Pass das passende `Translations`-Objekt und reichen es als Parameter durch — kein Modul-Singleton, keine Side-Effects (konsistent mit Spec §11.1 pure-functions-Geist). `EN` wird mit `const EN: Translations = {...}` typisiert, sodass der Compiler 1:1-Struktur zu DE erzwingt.

Locale-Match ist präfix-basiert: `hass.locale.language.startsWith('de')` → DE, alles andere → EN. Damit ist `de`, `de-DE`, `de-AT`, `de-CH` abgedeckt; jeder andere Wert (inkl. `undefined`) fällt auf EN zurück. Bestehende User mit HA-Locale ≠ `de` sehen ab dem Release eine geänderte Sprache — **Breaking Visual Change**, dokumentiert im Changelog (siehe §5.2).

### 0.0 TL;DR — Was der Planer NICHT tun darf

1. ❌ `src/engine/*` anfassen — Engine ist pure, kennt keine Strings.
2. ❌ `src/util/*` und `src/ha/*` mit i18n-Imports versehen — `util/` darf nichts importieren (ADR-0009 Layer-Zone), `ha/` ist HA-Boundary, beide lesen keine Strings.
3. ❌ Neues Config-Feld `display.language` oder ähnliches in `src/config/types.ts` einführen — Sprache kommt ausschließlich aus `hass.locale.language` (siehe §0.4 + §1.2).
4. ❌ Dynamic-Import (`await import('./en.ts')`) verwenden — Bundle-Strategie ist statisch (§1.4).
5. ❌ Modul-Singleton-Pattern (`let currentT = DE; export function setLang(l) {...}`) einführen — `T` wird als Parameter durchgereicht, niemals global gestateet.
6. ❌ `src/index.ts` (customCards-Registry) auf `resolveT` umstellen — Registry läuft vor hass-Available, Strings sind dort statisch englisch (siehe §1.3).
7. ❌ Bestehende `DE`-Objekte rename/move — `src/i18n/de.ts` bleibt unverändert, dient als Source-of-Truth für `type Translations = typeof DE`.
8. ❌ Reaktiven Sprach-Switch ohne hass-Update bauen — Lit-Reaktivität via `willUpdate` reicht; kein eigener Event-Bus.
9. ❌ Weitere Sprachen (FR, ES, …) hinzufügen — v1.x-Kandidat (siehe §6).
10. ❌ README-Übersetzung englisch — Doku bleibt deutsch in dieser Spec (siehe §6).
11. ❌ Eigene `Lang`-Type-Definition außerhalb `src/i18n/index.ts` — Single-Source.
12. ❌ `as any` / `as unknown` für `EN`-Konstruktion — wenn Typ nicht passt, ist Struktur falsch (siehe §1.1 Type-Safety).
13. ❌ Test-File für `de.ts` erstellen (`de.test.ts`) — bestehende Renderer/Editor-Tests verifizieren DE-Pfad bereits indirekt; nur EN-spezifische Tests neu (§3.2).
14. ❌ Editor-Schema (`ha-form`-Schema) Sprach-abhängig konstruieren — Editor liest dasselbe `T` wie die Card und gut.

Bei Konflikt zwischen Verbot und Plan-Schritt: STOP und nachfragen.

### 0.3 Konzept-Modell — Datenfluss-Pipeline

Wichtig damit der Planer versteht, **welcher** Schritt durch diese Spec berührt wird:

```
HA hass.states ─┐
                ├─► buildSystemState(config, hass) ─► engine.compute() ─► FlowResult
config         ─┘                                          (pure)              │
                                                                               │
hass.locale.language ─► langFromHass() ─► Lang ─► resolveT() ─► Translations ──┤
                          (NEU, §1.2)             (NEU, §1.1)                  │
                                                                               ▼
                                                              renderCard(layout, flow, T, ctx)
                                                                  (T wird als Parameter durchgereicht)
```

**Pflicht-Wissen:**

- Engine bleibt **vollständig sprachneutral** — keine `T`-Parameter dort, keine String-Outputs.
- `Translations` wird in `card.ts.willUpdate` aus `hass.locale.language` resolved und in `render()` an alle Renderer-/Helper-Aufrufe weitergereicht.
- `derive-display-consumers` (Config-Layer) bleibt sprachneutral — setzt `name: undefined` für die unassigned-Gruppe; der Renderer löst die Default-Bezeichnung gegen `T.nodes.unassignedGroup` auf.
- `customCards`-Registry (`src/index.ts`) läuft **vor** dem ersten `hass`-Update → Strings dort statisch englisch hardcoded (kein i18n-Lookup möglich).

### 0.4 Lit-Lifecycle (`card.ts` / `editor.ts`)

```
setConfig(yaml)     ─► validateConfig ─► this._config = ... (sync)
hass = newHass      ─► shouldUpdate(prev, new)
                         ├─ false: skip
                         └─ true: ↓
                       willUpdate():
                         this._lang = langFromHass(this.hass)   ← NEU
                         buildCardState → engine.compute → computeLayout
                                     Side-effects: this._flowResult = ..., this._layout = ...
                       render():
                         const T = resolveT(this._lang)         ← NEU
                         produces TemplateResult (PURE)
```

**Pflicht-Wissen:**

- `shouldUpdate` (ADR-0011) muss `locale.language`-Änderungen erkennen, sonst rendert die Card bei HA-Sprachwechsel nicht neu. Heutige Implementation vergleicht `hass.states` Sensor-IDs — für i18n muss zusätzlich `oldHass?.locale?.language !== this.hass?.locale?.language` als trigger gelten. (Konkrete Implementierung: Planer ergänzt eine Bedingung in der bestehenden `shouldUpdate`-Logik.)
- `_lang` ist `@state()`-Property (Lit-reaktiv) — Wechsel triggert re-render automatisch.
- `render()` bleibt PURE — `resolveT()` ist eine reine Lookup-Funktion, keine Side-Effects.

### 0.6 Engine-Warnings statt Throws (Konvention)

Bei dieser Spec wird die Konvention nicht aktiv getriggert (keine Engine-Änderung). Falls beim Refactor ein Edge-Case auftaucht (z.B. `T` ist `undefined` weil hass noch nicht da war): **niemals throwen**, sondern Fallback auf EN. `resolveT(lang)` und `langFromHass(hass)` sind so designt, dass beide Funktionen mit `undefined`-Eingaben sinnvolle Defaults liefern (siehe §1.1).

## 1. Sprach-Resolution

### 1.1 `src/i18n/index.ts` — Resolver-Modul

```ts
import { DE } from './de';
import { EN } from './en';

export type Translations = typeof DE;
export type Lang = 'de' | 'en';

/** Strukturelle hass-Sicht — kein HA-Type-Import, damit i18n-Layer-Zone gewahrt bleibt. */
type HassLocaleView = { locale?: { language?: string } } | undefined;

export function langFromHass(hass: HassLocaleView): Lang {
  const lang = hass?.locale?.language;
  if (typeof lang === 'string' && lang.toLowerCase().startsWith('de')) return 'de';
  return 'en';
}

export function resolveT(lang: Lang): Translations {
  return lang === 'de' ? DE : EN;
}
```

**Type-Safety:**

- `type Translations = typeof DE` macht DE zur Source-of-Truth für die Struktur.
- `src/i18n/en.ts` deklariert `export const EN: Translations = {...}` — fehlende Keys oder falsche Typen brechen `pnpm typecheck`.
- Funktionale Strings (`pluralize`, `pairingMissing`) sind in DE als Funktions-Typen typisiert (siehe `de.ts`); EN muss dieselben Signaturen liefern.

**Locale-Match (präfix-basiert):**

- `'de'`, `'de-DE'`, `'de-AT'`, `'de-CH'` → `'de'`
- `'en'`, `'en-US'`, `'en-GB'`, …, `'fr'`, `'es'`, `'ja'`, `undefined`, `''` → `'en'`
- `.toLowerCase()` ist defensiv (HA liefert üblicherweise lowercase, aber Belt-and-Suspenders).

### 1.4 Bundle-Strategie

- DE und EN sind **statisch** im Main-Bundle (`dist/custom-energy-flow-card.js`).
- Keine Dynamic-Imports (`await import('./en.ts')`).
- Erwarteter Bundle-Zuwachs: EN-Strings sind ~10–20 % kürzer als DE → geschätzt +0.8–1.2 KiB minified.
- Aktuell: 61.4 KiB (baseline metrics) → erwartet ~62.5 KiB → klar unter 64 KiB Budget (ADR-0022).
- Verifikation: `pnpm build` nach Implementation, gegen `BUNDLE_BUDGET_BYTES` in `scripts/kpi.mjs:29`.

## 2. Code-Änderungen

### 2.1 Caller-Pattern-Refactor

**Vorher (alle 9 Importer):**

```ts
import { DE } from '../i18n/de';
// ...
const label = DE.nodes.solar;
```

**Nachher:**

```ts
import type { Translations } from '../i18n';
// Im Renderer/Helper:
export function renderNode(ctx: RenderContext, T: Translations, node: LayoutNode) {
  // ...
  const label = T.nodes.solar;
}
```

**Wer ruft `resolveT()` auf?**

- `card.ts.render()`: `const T = resolveT(this._lang)` — einmal pro Render-Pass.
- `editor.ts.render()`: analog.
- `card.ts.willUpdate()`: setzt `this._lang = langFromHass(this.hass)`.
- `editor.ts.setConfig() / hass-Setter`: setzt `_lang` analog (Editor hat eigene LitElement-Reaktivität).

**Wer bekommt `T` als Parameter?**

- Alle Funktionen in `render/flow-renderer.ts`, `render/node-renderer.ts`, `render/battery-ring.ts`, `card-helpers.ts`, `editor-list-sections.ts`, die heute `DE` direkt importieren.
- `derive-display-consumers.ts` (Config-Layer) bekommt **kein** `T` — bleibt sprachneutral (siehe §2.2).

function walk(a: unknown, b: unknown, path = ''): void {
  if (typeof a === 'function') {
    expect(typeof b).toBe('function');
    return;
  }
  if (typeof a === 'string') {
    expect(typeof b).toBe('string');
    expect((b as string).length).toBeGreaterThan(0);
    return;
  }
  if (a && typeof a === 'object') {
    expect(b).toBeTypeOf('object');
    expect(b).not.toBeNull();
    for (const key of Object.keys(a)) {
      expect(b).toHaveProperty(key);
      walk(
        (a as Record<string, unknown>)[key],
        (b as Record<string, unknown>)[key],
        `${path}.${key}`,
      );
    }
  }
}

## 4. Dokumentation

### 4.1 ADR-0023 Neu — Stub für den Planer

Neuer ADR `docs/adr/0023-i18n-via-hass-locale.md`. **Stub unten 1:1 übernehmen** und in den ADR-Index (`docs/adr/README.md`) eintragen.

```markdown
# ADR-0023: i18n via HA-Locale (DE/EN) mit resolveT-Factory

- **Status:** accepted
- **Datum:** 2026-05-15
- **Entscheider:** @griebner

## Kontext und Problem

Hauptspec §1.4 listet Internationalisierung als bewussten v1.x-Kandidaten („trivial weil zentralisiert"). Für eine runde v1.0 soll Englisch als zweite Sprache verfügbar sein, damit die Card außerhalb des deutschsprachigen Raums (HACS-Reichweite) nutzbar ist.

Drei Architektur-Fragen:

1. Woher kommt die Sprache? Config-Feld oder HA-Locale?
2. Wie holen Caller das Translation-Objekt? Modul-Singleton oder Parameter-Durchreichung?
3. Wie wird die EN-Datei type-sicher 1:1 zur DE-Datei gehalten?

## Entscheidung

**Gewählt:**

1. **Sprache aus `hass.locale.language`** (präfix-basiert: `de*` → DE, alles andere → EN). Kein Config-Feld.
2. **`resolveT(lang)`-Factory** in `src/i18n/index.ts`. Caller halten `_lang` als `@state`, holen `T = resolveT(_lang)` pro Render und reichen `T` als Parameter an Sub-Module weiter. Kein Modul-Singleton.
3. **`type Translations = typeof DE`** als Source-of-Truth; `EN: Translations` erzwingt 1:1-Struktur via TypeScript-Compiler.
4. **Bundle-Strategie statisch**: beide Sprachen im Main-Bundle (~+1 KiB), kein Dynamic-Import.

### Positive Konsequenzen

- Kein User-Eingriff nötig — Card folgt HA-Sprache automatisch.
- Pure-functions-Geist (§11.1) bleibt erhalten — `T` ist Parameter, kein globaler State.
- TypeScript-Compiler verhindert struktur-Drift zwischen DE und EN.
- Bundle bleibt klar unter 64 KiB (ADR-0022).
- Weitere Sprachen (FR, ES, …) sind nach v1.0 trivial: zusätzliche Datei, `Lang`-Type erweitern, `resolveT`-Branch ergänzen.

### Negative Konsequenzen

- **Breaking Visual Change**: User mit HA-Locale ≠ `de` sehen ab Release Englisch. Dokumentiert im Changelog.
- Keine User-Override-Möglichkeit (kein `display.language`-Feld). Bei Bedarf v1.x-Patch.
- Sprach-Switch erfordert hass-Update (kein eigener Event-Bus) — in der Praxis kein Problem, da HA-Sprachwechsel typischerweise Reload triggert.

## Geprüfte Optionen

- **A — HA-Locale-driven + resolveT-Factory** (gewählt)
- **B — Config-Feld `display.language`** — mehr User-Kontrolle, aber Editor-Feld nötig, weniger automatisch
- **C — Modul-Singleton mit `setLang()`** — kürzer in Callern, aber stateful Module = Anti-Pattern (§11.1)
- **D — Dynamic-Import lazy per Sprache** — minimal kleineres Bundle, aber Async-Komplexität ohne Nutzen bei < 5 KiB Differenz

## Verlinkte Spec-Sektionen / Referenzen

- Subspec [`docs/specs/2026-05-15-en-i18n.md`](../specs/2026-05-15-en-i18n.md)
- Hauptspec §1.4 (Out-of-Scope, aufgelöst durch diese Entscheidung)
- ADR-0010 (Single-Source-Util-Modul — strukturell analog für `i18n/`)
- ADR-0011 (`shouldUpdate` über `hasChanged` — Locale-Vergleich wird ergänzt)
- ADR-0022 (Bundle-Budget 64 KiB — eingehalten)
- [HA Frontend: Locale](https://developers.home-assistant.io/docs/frontend/data/)
```

## 5. Migration und Breaking Changes

| `de`, `de-DE`, `de-AT`, `de-CH`   | DE wie bisher        | nein                                  |

Datenmapping bleibt 1:1, alle Sensoren und Edges identisch. Nur User-facing-Strings sind betroffen.

### 5.2 Was _nicht_ bricht

- Config (`type`, `solar`, `battery`, `consumers`, `display`) bleibt strukturell unverändert.
- HA-Entity-Mapping (kein Sensor-Rename).
- Engine-Output (Flow-Werte, Warnings).
- Animation, Layout, Theme, `::part()`-Hooks.
- Consumer-Grouping (ADR-0016).
- `aria-label`-Struktur (nur Inhalt lokalisiert).

### 5.4 Operationelle Aspekte

**Rollback:** rein additiv. Rollback = `git revert` der vier Commits aus §2.3. Configs bleiben kompatibel — kein Daten-Rollback nötig.

**Pre-Release-Gate (ADR-0012):** Headless-Smoke-Test rendert Card mit Mock-hass. Falls Mock-hass kein `locale`-Feld setzt, fällt `langFromHass` auf EN — Smoke-Test bleibt grün. Falls das Smoke-Test-Asset Strings hardcoded prüft, müssen diese auf EN angepasst oder sprachneutral (z.B. nur strukturelle Checks via Selector) gemacht werden. **Planer-Pflicht:** beim ersten Test-Lauf prüfen.

**Performance:** kein Engine-Eingriff, kein Layout-Eingriff. `resolveT(lang)` ist O(1) Lookup. Bundle-Overhead: ~+1 KiB minified.

**Pflege-Aspekt:** zukünftige String-Änderungen MÜSSEN beide Files (`de.ts`, `en.ts`) updaten. TypeScript-Compiler fängt strukturelle Drift; Wert-Drift („englischen String vergessen zu aktualisieren") ist Reviewer-Pflicht (§4.5 Convention-Eintrag).

## 6. Out of Scope

- Weitere Sprachen (FR, ES, IT, …) — v1.x-Kandidat. Pattern ist nach v1.0 trivial erweiterbar: zusätzliche Datei, `Lang`-Type erweitern, `resolveT`-Branch.
- Config-Override für Sprache (`display.language`) — v1.x-Kandidat falls Bedarf besteht.
- Dynamic-Import / Lazy-Loading der Sprachdateien — Bundle-Budget reicht statisch.
- README-/Doku-Übersetzung — bleibt deutsch in dieser Spec. Internationale README ist v1.x-Kandidat.
- Übersetzung der `customCards`-Registry — statisch englisch, kein i18n-Lookup möglich vor hass-Available.
- Default-Knoten-Namen im Editor („Solar 1", „Battery 1"): bestehende `name`-Override-Logik bleibt; auto-generierte Defaults gibts nicht.
- Reaktiver Sprach-Switch ohne hass-Update — Lit-Reaktivität via `willUpdate` reicht.
